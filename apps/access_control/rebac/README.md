# ReBAC

## 개요

ReBAC(Relationship-Based Access Control)는 사용자와 리소스 간 관계를 기준으로 접근 권한을 계산하는 모델이다.

## 쇼핑몰에서 보는 이유

쇼핑몰은 사용자 역할만으로는 부족하고, "이 판매자의 직원인지", "이 주문의 구매자인지", "이 브랜드 운영팀에 속해 있는지" 같은 관계가 중요하다.

## 주요 관계 예시

- `buyer-of`
- `seller-of`
- `staff-of-seller`
- `manager-of-seller`
- `owner-of-brand`
- `operator-of-store`

## 쇼핑몰 권한 예시

- 주문 상세 조회는 `buyer-of(order)` 또는 `seller-of(orderItem)` 관계가 있을 때 허용한다.
- 판매자 직원은 `staff-of-seller` 관계가 있는 스토어의 상품만 수정할 수 있다.
- 브랜드 운영자는 `owner-of-brand` 관계가 있는 브랜드 상품만 승인 요청할 수 있다.
- 리뷰 답변은 해당 상품을 판매한 판매자와의 관계가 있을 때만 허용한다.

## React + NestJS 스터디 예시

프론트엔드는 현재 사용자와 스토어, 브랜드, 주문의 관계에 따라 메뉴를 다르게 보여준다.
백엔드는 관계 그래프 조회 또는 관계 테이블 탐색으로 접근 가능 범위를 계산한다.

## 데이터 흐름

1. 사용자가 주문, 상품, 스토어에 접근한다.
2. 서버가 사용자-리소스 관계를 조회한다.
3. 필요하면 스토어, 브랜드, 셀러 조직까지 확장 탐색한다.
4. 판정 결과를 반환한다.

## 적합한 사례

- 스토어, 브랜드, 공급사가 함께 연결된 마켓플레이스
- 판매자 조직 단위 권한이 중요한 쇼핑몰
- 관계 기반 공유가 필요한 운영 도구

## 한계

- 관계 모델링이 복잡하다.
- 그래프 탐색 비용과 캐싱 전략이 중요하다.
