# ABAC

## 개요

ABAC(Attribute-Based Access Control)는 사용자, 리소스, 액션, 환경 속성을 조합해서 접근 허용 여부를 계산하는 모델이다.

## 쇼핑몰에서 보는 이유

쇼핑몰은 같은 `seller` 역할이라도 자기 상품만 수정 가능하거나, 정산 마감일 이후에는 환불 승인을 막아야 하는 식의 문맥 조건이 많다.

## 주요 속성 축

- Subject: 사용자 유형, 판매자 ID, 담당 조직, 인증 등급
- Resource: 상품 소유 판매자 ID, 주문 상태, 환불 금액, 리뷰 신고 여부
- Action: `read`, `update`, `approve`, `reject`, `export`
- Environment: 영업 시간, 관리자 IP, 해외 접속 여부, 위험 거래 점수

## 쇼핑몰 정책 예시

- 판매자는 `resource.sellerId === subject.sellerId`인 상품만 수정할 수 있다.
- 고객센터 상담원은 `order.status !== delivered`인 주문만 취소 접수할 수 있다.
- 정산 담당자는 `environment.ipTrusted === true`일 때만 정산 파일을 다운로드할 수 있다.
- 고액 환불은 `refund.amount >= 300000`이면 팀장 승인 속성이 추가로 필요하다.

## React + NestJS 스터디 예시

프론트엔드는 주문 카드마다 `canCancel`, `canRefundApprove`, `canEditProduct` 같은 계산 결과를 표시한다.
백엔드는 정책 평가 서비스가 사용자 속성과 리소스 속성을 조합해서 허용 여부를 판정한다.

## 데이터 흐름

1. 사용자가 상품, 주문, 환불 리소스를 요청한다.
2. 서버가 사용자 속성과 리소스 속성을 로드한다.
3. 정책 엔진이 액션과 환경 조건을 함께 평가한다.
4. 결과를 응답하고 프론트엔드가 UI를 조정한다.

## 적합한 사례

- 판매자별 정책이 다른 오픈마켓
- 위험 거래 탐지와 운영 정책이 많은 커머스
- 세밀한 문맥 기반 권한 제어가 필요한 서비스

## 한계

- 정책 수가 많아지면 가독성이 떨어진다.
- 디버깅과 테스트 설계가 RBAC보다 어렵다.
